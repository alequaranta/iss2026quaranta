FireflySynch B

Introduction

Goal: Gestione di 3 lucciole, presenza di un sonar in grado di rilevare ostacoli

Requirements

-	Costruire il simulatore di un sistema costituito da 3 lucciole. Inizialemente, le lucciole lampeggiamo ciascuna con una
	frequenza (costante) propria.
	
	B) Il sistema è dotato anche di un sonar . Quando il sonar rileva un ostacolo a una distanza minore di una distanza
	prefissata DMIN, le lucciole devoono lampeggiare in modo sincronizzato. Quando la distanza dell’ostacolo
	(ri)supera DMIN, le lucciole tornano a lampeggiare in modo random.

Requirement analysis

-	PREMESSA: Per il seguente sistema si fa riferimento a quanto già analizzato per il requisito A
	
-	In accordo con il commitente il sistema è descritto secondo modello Qak.

-	Nel modello precedente abbiamo capito che:
		> La lucciola è modellata come un QActor in quanto dotata di comportamento proprio.
		> L'unico suo comportamento è quello di accendersi e spegnersi ad una determinata frequenza.
		> Le lucciole aggiornano la loro frequenza secondo il pattern Observer; la comunicazione avviene tramite:
		
			Dispatch synch : changed(SOURCE,TIME)
				
		> Le lucciole osservano e reagiscono ai cambiamenti:
		
			observeResource firefly_queen msgid synch
		
-	Il nuovo modello non richiede più la presenza di una lucciola regina.
	Il sistema presenta un sonar in grado di rilevare distanze da esso.
	>	Comportamento del sonar: 
		> 	quando rileva un ostacolo a una distanza minore di una distanza
			prefissata DMIN; segnala alle lucciole di allineare la frequenza di lampeggiamento
		>	quando risupera la distanza DMIN; segnala alle lucciole frequenza libera

-	Viene sviluppata una prima versione mock del sonar.
	In maniera random il sonar rileva l'ostacolo.
	
-	Si sviluppa un modello che segue ancora una volta il pattern observer:
	> Il sonar aggiorna il proprio stato e lo comunica tramite:
		
				Dispatch synch : changed(SOURCE,SYNCH,TIME)
			
	SYNCH	vale = 1 se rileva un ostacolo
			vale = 0 se risupera l'ostacolo
	TIME	frequenza da mantenere
	
	> NOTA: in alternativa è possibile ricondursi al modello in cui una lucciola regina determina la frequenza delle altre.
	> Le lucciole osservano e reagiscono ai cambiamenti:
		
				observeResource sonar_mock msgid synch
			
-	Se si vollesse sincronizzare le lucciole, oltre che in frequenza, anche in fase, sarebbe necessario
	allineare le lucciole su un tempo assoluto. L'ipotetica implementazione è affidata al progettista.

Problem analysis

	fireflysynch.qak

-	Gli attori che fungono da lucciola possono essere generati dinamicamente da un "creator".
	
		QActor firefly context ctxfirefly dynamicOnly{
	

-	Secondo il pattern Observer, le lucciole hanno il compito di reagire ai cambiamenti dello stato osservato:
		
						observeResource sonar_mock msgid synch
			
	Se il sonar ha raggiunto la distanza minima dall'ostacolo, le lucciole si allineano sulla frequenza segnalata:
	
		onMsg( synch : changed(SOURCE,SYNCH,TIME) ){
		        [#
		        	var s = payloadArg(1)
		        #]
		        
		        if[# s == "1" #]{
		        	 [# var t = payloadArg(2)
		            	Timer = t.toLong() #]     	
		        }
		
-	La simulazione del sonar (sonar_mock) rileva ogni 5 secondi una nuova distanza e 
	aggiorna il proprio stato di conseguenza.
	
		State eval{
		  
		  		delay 5000
		  		[# D = java.util.Random().nextInt(10, 150) #]
				
				println("$name | new distance: $D") color yellow
				
				if [# D < DMIN #]{
					[# SYNCH = 1 #]
					println("$name | SYNCH") color yellow
					updateResource [# "$name,$SYNCH,$Timer" #]
				}
	
-	NOTA: In una versione più realistica il sonar emette un messaggio di tipo emit verso il contesto in cui
	risiedono le lucciole. I cambiamenti sul modello sarebbero MINIMI: firefly al posto elaborare un Dispatch elabora un Event.
	Punto chiave rimane il formato dei messaggi scambiati.
	

	griddisplay.qak
	
-	L'output delle lucciole è reso visibile all'utente finale tramite il sistema griddisplay,
	pensato per il gioco Conway Game Of Life. La GUI raggiungibile via Browser Web rappresenta 
	una griglia 20x20.
	
		ExternalQActor griddisplay context ctxgrid
		
-	Il formato dei messaggi è il seguente:	
	
		Dispatch cellstate : cellstate(X, Y, COLOR)
	
	X, Y coordinate sulla griglia
	COLOR vale = 1 se accesa, = 0 se spenta
